Cloudflare Pages 和 VPS 部署怎么选

很多小团队不是不会部署，而是在一开始把部署边界选错了：静态页面被放到 VPS 上长期维护，或者需要持久进程的后台硬塞进静态托管。正确的问题不是“哪个更先进”，而是“你的应用需要什么运行时、状态和控制面”。

一句话结论

能静态化、无长连接、无本机持久进程、无需复杂后台运维的内容，用 Cloudflare Pages。需要自定义运行时、数据库私网访问、队列、长任务、WebSocket、后台管理系统或多服务编排，用 VPS 或容器化平台。

实战上最稳的默认策略是：官网、落地页、文档、活动页优先放到 Pages；业务后台、SaaS、爬虫任务、支付回调处理、复杂 API 服务优先放到 VPS；前端静态化 + API 独立部署是多数小团队的成本最优解。

Cloudflare Pages 适合什么

Pages 的核心价值不是“免费”，而是把发布链路变短：Git 推送、预览环境、全球 CDN、HTTPS 和回滚能力被平台接管。对于小团队来说，这会直接减少运维摩擦。

品牌官网 / 落地页

这类页面主要承载信任、展示、获客和转化。它们应该加载快、部署快、回滚快，不应该让你每天维护服务器。

文档站 / 知识库

Markdown、静态站生成器和纯前端搜索很适合 Pages。内容更新走 Git，审稿和发布记录天然保留。

活动页 / H5

短周期页面最怕“上线流程太重”。Pages 适合快速生成预览链接，让业务方先看再发。

不要把 Pages 当成完整后端。表单、登录、支付、管理后台、消息队列和长任务可以接入外部服务或 Workers，但如果复杂度明显上升，就应该拆出真正的后端。

VPS 适合什么

VPS 的价值是控制权。你可以决定进程怎么启动、日志放哪里、数据库怎么备份、nginx 怎么路由、证书怎么签、什么时候重启服务。代价是你要对这套系统负责。

  • 后台系统：例如 RuoYi、Vue Admin、Django Admin、Laravel 后台，需要 API、数据库、权限和文件上传。
  • Node / Java / Python 服务：需要常驻进程、任务队列、定时任务、WebSocket 或内部脚本。
  • 多租户 SaaS：需要域名路由、租户隔离、数据库迁移、日志审计和灰度发布。
  • 自定义运维：需要按自己的方式做备份、回滚、安全加固、监控告警和成本控制。

如果你选择 VPS，最低限度也要有一份可重复执行的部署脚本。手工 SSH 上去改文件不是部署体系，只是临时操作。

最常见的混合架构

多数项目不需要二选一。更合理的做法是把“展示层”和“业务层”分开，让静态资源走 CDN，让有状态业务留在受控后端。

用户浏览器
  -> Cloudflare Pages: 官网 / 前端 / 文档 / 活动页
  -> API 域名: api.example.com
      -> VPS / Docker / Node / Java / Python
      -> 数据库 / Redis / 文件存储

这种结构的好处很实际：

  • 首页和文档加载快，且不受后端重启影响。
  • 后端可以独立扩展、重启、回滚，不必拖着前端一起发布。
  • 域名和证书边界清晰，出现问题时排查面更小。
  • 以后迁移云厂商或换框架时，前端入口可以保持稳定。

上线前检查清单

不管选 Pages 还是 VPS，上线前都要把“怎么发布、怎么回滚、怎么发现问题”写清楚。下面这份清单可以直接复用。

  1. 域名：确认主域名、www、API 子域名分别指向哪里，避免 A 记录、CNAME 和代理状态混乱。
  2. HTTPS：确认访问链路只有一种跳转方向，例如统一到 https://example.com。
  3. 构建：记录构建命令、输出目录、环境变量和 Node 版本。
  4. 发布：不要依赖口头流程。至少写出 build -> deploy -> verify 三步。
  5. 回滚：Pages 用历史部署回滚；VPS 至少保留上一版目录快照或镜像标签。
  6. 验证：准备 3 到 5 个关键 URL，发布后自动或手动 curl 检查状态码。
  7. 日志：知道访问日志、应用日志和错误日志分别在哪里。
  8. 备份：只要有数据库，就必须明确备份频率、保留时间和恢复演练方式。
天智工坊自己的建站工具链里，`site-bootstrap` 解决的正是 VPS 路线的重复劳动：DNS、rsync、nginx、证书、验证和回滚尽量脚本化。它不替代平台，但能把小团队最容易出错的手工步骤收束起来。

最后的判断标准

如果一个方案让你上线更快，但排查问题更慢，它只适合短期页面。如果一个方案让你控制力更强，但每次发布都要靠记忆，它也不成熟。小团队的部署架构应该满足三个条件：能复现、能回滚、能解释。

因此，默认从 Cloudflare Pages 开始没有问题；当你开始需要常驻进程、数据库、后台任务和更细的运维控制时，再把业务层迁到 VPS。不要为了“看起来专业”提前复杂化，也不要为了“省事”把真正的后端塞进静态托管。
